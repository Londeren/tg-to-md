## Зачем

Два связанных изменения, которые решают один и тот же пробел — прозрачность формата для LLM-читателя:

- **`forwarded_from` сейчас выкидывается.** В личных чатах часто встречаются пересланные сообщения (мемы, цитаты из других диалогов). Без маркера авторства LLM не может отличить оригинальное сообщение собеседника от пересланного и теряет важный сигнал: «это не собственная реплика, а ретрансляция».
- **Накопилось много спец-обозначений** (`###`, `↩`, `[emoji×N]`, `🧩`, `🖼️`, `📎`, `🎤`, `📌`, `📞`), и читатель (человек или LLM) вынужден выводить их семантику из контекста. Явный словарь в шапке файла снимает эту неопределённость и стоит буквально несколько строк.

Оба изменения затрагивают только `src/render.js` и `src/pipeline.js` — без правок парсера и CLI.

## Вне области

- Поле `forwarded_from_id` — игнорируется, не рендерится (на него невозможно сослаться внутри файла, в отличие от `reply_to_message_id`).
- Различение «переслано от пользователя» vs «переслано из канала» — не делаем. Структурно `forwarded_from` в обоих случаях одна строка, поведение одинаковое.
- Локализация легенды — строка статическая, на английском (языковая нейтральность, консистентно с остальной meta-обвязкой в файле).
- `edited`, `via_bot` — по-прежнему не рендерим. Их добавление — отдельный вопрос.

## Изменения

### 1. Маркер `forwarded_from` в заголовке сообщения

Был формат хедера:

```
### #<id> — <from> · <date>[ · ↩ #<reply_to_message_id>][ · [<reactions>]]
```

Стал — между `<date>` и `↩` вставляется опциональный сегмент `↪ <forwarded_from>`:

```
### #<id> — <from> · <date>[ · ↪ <forwarded_from>][ · ↩ #<reply_to_message_id>][ · [<reactions>]]
```

Разделитель — тот же `·`, что и остальные сегменты хедера.

**Порядок сегментов зафиксирован** (важно для тестов и предсказуемости для LLM): `from`, `date`, `↪ forwarded_from`, `↩ reply_to`, `[reactions]`. Логика: `date` «прилипает» к автору (кто-когда), далее атрибуты источника — сначала внешний (`↪`), затем внутренний (`↩`), реакции всегда в конце.

Пример:

```
### #152 — Sergey Lebedev · 2014-07-29T12:31:20 · ↪ Саня Барабаш · ↩ #149 · [👍×2]
так блин, чо хоть вы
```

**Правила эмита сегмента:**

| Значение `forwarded_from` | Поведение |
|---|---|
| непустая строка (включая `"Hidden User"`) | рендерим `↪ <строка>` как есть |
| `null` | сегмент не эмитим |
| `""` (пустая строка) | сегмент не эмитим |
| поля нет | сегмент не эмитим |

Поле `forwarded_from_id` полностью игнорируется.

Для сервисных сообщений (`type: "service"`) поле `forwarded_from` в экспорте не встречается; ветка условно недостижима, но реализация должна вести себя корректно даже если оно появится — пусть работает общий путь (проверка непустой строки).

### 2. Блок легенды в начале файла

Один статический блок, эмитится один раз в самом начале выходного файла — **до шапки первого чата**.

**Содержимое** (финальная форма, зафиксированная как константа в `src/render.js`). Ниже показан итоговый фрагмент выходного файла — включая разделитель, который эмитит `pipeline.js` (см. § 3):

```markdown
# Legend

- `### #<id>` — message id
- `↩ #<id>` — reply to message
- `↪ <name>` — forwarded from
- `[emoji×N, …]` — reactions (🧩 = custom emoji group)
- 🖼️ photo · 📎 file · 🎤 voice/video note · 📌 pin · 📞 call

---

```

Сама `renderLegend()` возвращает только часть до разделителя — заканчивается одиночным `\n` после строки `- 🖼️ photo · 📎 file · …`. Разделитель `\n---\n\n` добавляет `pipeline.js`, тот же, что используется между чатами в bulk-экспорте.

**Когда рендерится:**

- Всегда, если на вход попал хотя бы один чат (single-chat и bulk — одинаково).
- Если `chatCount === 0` (пустой или некорректный вход) — легенду **не** эмитим, чтобы не выдавать файл, состоящий только из легенды. Поведение CLI при пустом входе не меняется (возврат `exit code` и stderr — как есть).

### 3. Архитектурные изменения

#### `src/render.js`

- Новая exported `renderLegend()`. Чистая функция без аргументов, возвращает константную строку: markdown-блок `# Legend` с пятью буллетами, заканчивающийся одиночным `\n` после последней строки `- 🖼️ photo · 📎 file · …`. Разделитель `---` и окружающие переводы строк в строку не входят — их добавляет `pipeline.js` (см. ниже), по аналогии с тем, как он добавляет разделитель между чатами.
- `renderMessageHeader(msg)` — после блока с датой и до блока с `↩ #reply_to_message_id` добавить сегмент для `forwarded_from`, если он непустая строка.

#### `src/pipeline.js`

- В `renderExport(parseResult, write)`, до цикла по чатам, проверить: если хотя бы один чат будет отрендерен, вызвать `write(renderLegend())`, затем `write("\n---\n\n")`.
- Существующий emitter разделителя между чатами (`\n---\n\n`) не трогаем — логика «легенда → `---` → чат1 → `---` → чат2 → …» получается естественно.
- Возвращаемая структура `{ chatCount, rendered, skippedTotal, firstMeta }` не меняется. Счётчики не инкрементируются от легенды.

- Пустой экспорт (`chatCount === 0`): легенда не эмитится.

## Риски и что проверить при реализации

- **Порядок сегментов в хедере** — единственный реальный источник расхождений с текущим поведением. Покрыть тестом именно комбинацию всех четырёх опциональных сегментов (`↪`, `↩`, `[reactions]`) в одном сообщении.
- **Backpressure в pipeline.** Вызов `write(renderLegend())` подчиняется тем же правилам, что остальные write'ы в пайплайне — если write возвращает `false`, дождаться `drain`. На практике легенда — маленькая строка, риск минимальный, но корректность обработки сохранить.
- **Service-сообщения и `forwarded_from`.** Реализация должна идти через общую ветку в `renderMessageHeader` и не требовать специальной проверки на `msg.type === "service"`. Правило простое: сегмент эмитим, если `msg.forwarded_from` — непустая строка, независимо от типа сообщения.
- **Константа легенды** — держать прямо в `src/render.js` как локальную константу. Не выносить в отдельный файл (формат статичен, частота изменений — раз в несколько рефакторов).

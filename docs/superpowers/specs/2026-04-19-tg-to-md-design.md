# tg-to-md — дизайн

**Дата:** 2026-04-19
**Статус:** черновик на согласовании

## Цель

CLI-утилита на Node.js, которая берёт экспорт Telegram-чата в формате JSON (как `result.json` / `partita-iva.json`) и превращает его в один Markdown-файл, оптимизированный для последующей подачи в LLM (поиск по содержимому, саммаризация, вопросы-ответы).

## Требования

1. Запуск через `npx` без предварительной установки зависимостей пользователем.
2. Поддержка входных файлов любого размера (десятки и сотни мегабайт, гигабайты) без падений. Память не должна расти пропорционально размеру входа.
3. Один выходной `.md` файл — удобно для grep, загрузки в LLM, хранения в git.
4. Формат вывода — читаемый человеком и понятный LLM: у каждого сообщения есть уникальный идентификатор, видны автор, время и связи между сообщениями.

## Нефункциональные требования

- Без шага сборки — исходный JavaScript запускается напрямую. Это необходимо для того, чтобы `npx tg-to-md` работал без компиляции у пользователя.
- Минимум runtime-зависимостей (сейчас: только `stream-json`).

## CLI

```
npx tg-to-md <input.json> [output.md]
```

- `input.json` — обязателен.
- `output.md` — опционален. Если не указан, берём путь `input`, заменяем расширение на `.md` и пишем рядом. Пример: `partita-iva.json` → `partita-iva.md` в той же папке. Если у `input` нет расширения, просто дописываем `.md`.

Если файл с таким output-путём уже существует — перезаписываем (стандартное поведение `fs.createWriteStream`, отдельно не предупреждаем).

Выход из процесса с ненулевым кодом при отсутствии `input`-аргумента, несуществующем входном файле, ошибке парсинга или ошибке записи.

## Формат вывода

### Шапка файла

В самом начале `.md` — один блок с метаданными чата:

```markdown
# Partita iva 🇮🇹

Экспорт Telegram-чата. ID: 1561962579.

---
```

Счётчик сообщений в шапку не пишем — он неизвестен на момент записи шапки (парсим потоково).

### Формат одного сообщения

```markdown
### #25702 — Olga Valeeva · 2026-04-16 10:29 · ↩ #25701

Здравствуйте! А вы на каком режиме? Подробнее в [этой статье](https://example.com).
```

Поля заголовка:
- `#<id>` — идентификатор сообщения из JSON, всегда первый. По нему LLM ищет сообщение по ссылке `↩ #...`.
- `<автор>` — поле `from`. Если `from` отсутствует (редкий случай), используем `from_id`.
- `<дата> <время>` — `date` из JSON, отформатированный как `YYYY-MM-DD HH:MM` (секунды обрезаем). Значение берётся как есть, без конвертации таймзон — Telegram экспортирует локальное время экспортировавшего.
- `↩ #<reply_to_message_id>` — только если у сообщения есть `reply_to_message_id`. Сам текст сообщения-цели не дублируем: LLM при необходимости найдёт его по ID в том же файле.

Между сообщениями — одна пустая строка.

### Правила рендеринга тела

| Источник | Правило |
|---|---|
| service-сообщения (`type: "service"` — pin, join, invite, remove, boost, migrate) | **пропускаем полностью** |
| `custom_emoji` в text_entities | пропускаем |
| `plain` | текст как есть |
| `bold`, `italic`, `underline`, `strikethrough`, `spoiler`, `code`, `pre` | только текст, форматирование убираем |
| `blockquote` (в тексте сообщения) | рендерим как `> ` в начале каждой строки блока |
| `link` (голый URL) | оставляем URL как есть |
| `text_link` (ссылка с видимым текстом) | `[текст](url)` |
| `mention` (@username) | plain-текст `@username` |
| `hashtag` | plain-текст `#tag` |
| `email`, `phone`, `mention_name` | plain-текст |
| стикер (поле `sticker_emoji` присутствует) | тело сообщения — сам эмодзи стикера; префиксы не применяем |
| фото (поле `photo` присутствует) | префикс 🖼️ перед текстом |
| голосовое (`media_type: "voice_message"` или `"video_message"` — круглое видео) | префикс 🎤 |
| видеофайл (`media_type: "video_file"`) | префикс 📎 |
| прочий файл (поле `file` без специального `media_type`) | префикс 📎 |
| `reactions` | пропускаем |
| `edited` | пропускаем (не отмечаем) |
| `forwarded_from` | пропускаем (не отмечаем) |

Порядок детекции медиа (первый сработавший выигрывает): стикер → голосовое → фото → видеофайл → прочий файл. Если ни один не сработал — префикса нет.

Замечания по реальным данным в `partita-iva.json`: 807 фото (поле `photo`, без `media_type`), 6 видеофайлов (`media_type: "video_file"`), 1 стикер (`sticker_emoji` + `media_type: "sticker"`), 50 прочих файлов. Голосовых в этом экспорте нет, но тула должна их корректно помечать, если они встретятся в других экспортах.

Если после применения правил тело пустое (например, пришёл только `custom_emoji`), сообщение всё равно пишем с пустым телом — чтобы не было дыр в нумерации (важно для reply-ссылок).

## Архитектура

Три модуля, по одному ответственному на модуль:

```
tg-to-md/
├── bin/tg-to-md.js       # CLI-обвязка
├── src/
│   ├── parser.js         # streaming JSON → async iterable сообщений
│   └── render.js         # чистая функция renderMessage(msg) → string | null
├── test/
│   ├── render.test.js
│   └── fixtures/         # маленькие JSON-сэмплы и эталонные .md
├── package.json
└── README.md
```

### `bin/tg-to-md.js`

- shebang `#!/usr/bin/env node`
- парсит `process.argv`, валидирует наличие input/output
- открывает `fs.createWriteStream(output)`
- пишет шапку (читает из JSON поля `name` и `id` — они лежат до массива `messages`)
- итерируется по сообщениям из `parser`, для каждого вызывает `renderMessage`, пишет результат в стрим
- обрабатывает ошибки: печатает в stderr, выходит с кодом 1

### `src/parser.js`

- использует `stream-json` + `stream-json/streamers/StreamArray` для потокового чтения массива `messages`
- дополнительно вытягивает `name` и `id` из верхнего уровня до начала массива (pick + через pipeline)
- экспортирует async-функцию `parseTelegramExport(inputPath) → { meta, messages }`, где `messages` — async iterable

### `src/render.js`

- `renderHeader(meta) → string` — шапка файла
- `renderMessage(msg) → string | null` — null означает «пропустить» (для service-сообщений)
- внутренние хелперы: `renderEntities(entities)`, `formatDate(iso)`, `mediaPrefix(msg)`
- **чистая функция** — никаких побочных эффектов, полностью юнит-тестируемая

## Потоковый путь данных

```
input.json
  └─> fs.createReadStream
        └─> stream-json pipeline
              └─> async iterator: сообщение за сообщением
                    └─> renderMessage(msg)
                          └─> writeStream.write(...)
                                └─> output.md
```

Память, используемая процессом, остаётся примерно постоянной независимо от размера входа — в памяти одновременно находится только одно сообщение плюс фиксированные буферы потоков. Write-поток поддерживает backpressure (ждём `drain`, если буфер заполнен).

## Обработка ошибок

- Несуществующий input → stderr + exit 1.
- Битый JSON → stderr с сообщением от stream-json + exit 1. Частичный вывод может остаться в output — не подчищаем (пользователь удалит сам).
- Ошибка записи (нет места, нет прав) → stderr + exit 1.
- Неизвестный тип entity → рендерим `text_entities[i].text` как plain (fallback, чтобы не терять контент).

## Тесты

Фреймворк: встроенный `node --test` (без внешних зависимостей).

**Один smoke-тест «от входа до выхода».** Утилита проста (парсер + чистый рендер), отдельные юнит-тесты на каждую ветку избыточны. Вместо них — одна минимальная синтетическая фикстура, покрывающая все значимые ветки рендера, и сравнение с эталонным выводом.

Файлы:
- `test/fixtures/sample.json` — валидный Telegram-экспорт на ~10 сообщений, собранных вручную для покрытия веток:
  1. Обычное сообщение.
  2. Сообщение с `reply_to_message_id` → заголовок содержит `↩ #<id>`.
  3. Сообщение с `text_link` → `[текст](url)`.
  4. Сообщение с `link` (голый URL) → URL как есть.
  5. Сообщение с `blockquote` в `text_entities` → `> `.
  6. Сообщение с полем `photo` → префикс 🖼️.
  7. Сообщение с `file` без `media_type` → префикс 📎.
  8. Сообщение с `media_type: "voice_message"` → префикс 🎤.
  9. Сообщение со `sticker_emoji` → эмодзи как тело.
  10. service-сообщение → пропущено.
  11. `custom_emoji` внутри текста → пропущен, остальной текст сохранён.
  12. `bold`/`italic` → форматирование убрано, plain-текст остался.
- `test/fixtures/sample.expected.md` — эталонный вывод, хранящийся в git. Служит одновременно тестом и документацией формата.

Сам тест в `test/smoke.test.js`: прогоняет `parser` + `render` на `sample.json`, сравнивает побайтно с `sample.expected.md`. Диф при падении даёт понятный feedback.

Тест реально-большого файла (`partita-iva.json`) из автоматических тестов исключён — файл не в git, использовать для ручной проверки при разработке.

## Зависимости

**runtime:**
- `stream-json` (streaming JSON parser)

**dev:**
- нет (тесты через встроенный `node --test`)

## package.json

Ключевые поля:
- `"name": "tg-to-md"`
- `"version": "0.1.0"`
- `"type": "module"` (ES modules)
- `"bin": { "tg-to-md": "bin/tg-to-md.js" }`
- `"files": ["bin", "src", "README.md"]`
- `"engines": { "node": ">=20" }`
- `"scripts": { "test": "node --test", "start": "node bin/tg-to-md.js" }`

## Что умышленно не делаем (YAGNI)

- Не рендерим реакции, редактирования, пересылки — пользователю не нужны.
- Не делаем деление на несколько файлов — один файл требуется явно.
- Не поддерживаем разные форматы вывода (JSON Lines, HTML) — не требуется.
- Не пишем на TypeScript — чтобы работал `npx` без сборки.
- Не используем CLI-фреймворки вроде `commander` — два аргумента проще разобрать вручную.
- Не поддерживаем чтение из stdin / запись в stdout — только именованные файлы.
